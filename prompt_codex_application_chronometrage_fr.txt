## Logique principale du chronométrage

L’application doit avoir **un chronomètre principal unique**.

Comportement obligatoire :

* le bouton START démarre le chronomètre principal,
* quand l’opérateur clique sur un participant, l’application enregistre le temps courant pour ce participant,
* ce clic **ne doit jamais arrêter** le chronomètre principal,
* le chronomètre doit continuer à tourner pour les participants suivants,
* seul le bouton STOP arrête complètement le chronomètre principal.

C’est donc une logique de **split / temps d’arrivée**, pas une logique d’arrêt complet à chaque participant.

Exemple :

* START
* le temps tourne
* clic sur participant A → enregistre son temps
* le chronomètre continue
* clic sur participant B → enregistre son temps
* le chronomètre continue
* etc.

## Logique de notation

Tu dois :

1. lire les fichiers texte contenant les barèmes dans le dossier du projet,
2. structurer ces barèmes de manière exploitable,
3. associer un barème à une section et à une activité,
4. calculer automatiquement la note selon le temps ou la valeur enregistrée,
5. permettre une évolution future des barèmes.

L’application doit supporter plusieurs types de notation :

* notation selon le temps,
* notation selon une distance / hauteur / longueur,
* notation type validé / non validé,
* structure extensible si d’autres cas apparaissent dans les fichiers.
